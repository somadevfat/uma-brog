# スキルクリエイター (Skill Creator)

このスキルは、効果的なスキルを作成するためのガイダンスを提供します。

## スキルについて

スキルは、専門知識、ワークフロー、およびツールを提供することで、Claudeの機能を拡張するモジュール化された自己完結型のパッケージです。特定のドメインやタスクのための「オンボーディングガイド」と考えてください。これらは、Claudeを汎用的なエージェントから、モデルが完全には持ち得ない手続き的知識を備えた専門的なエージェントへと変貌させます。

### スキルが提供するもの

1. 専門的なワークフロー - 特定のドメイン向けのマルチステップの手順
2. ツールの統合 - 特定のファイル形式やAPIを扱うための指示
3. ドメインの専門知識 - 企業独自の知識、スキーマ、ビジネスロジック
4. 整理されたリソース - 複雑で繰り返しの多いタスクのためのスクリプト、参照資料、アセット

## 核となる原則

### 簡潔さが重要

コンテキストウィンドウは公共の利益です。スキルは、システムプロンプト、会話履歴、他のスキルのメタデータ、および実際のユーザーリクエストなど、Claudeが必要とする他のすべてのものとコンテキストウィンドウを共有します。

**デフォルトの前提：Claudeはすでに非常に賢い。** Claudeがまだ持っていないコンテキストのみを追加してください。情報の各断片に対して、「Claudeは本当にこの説明を必要としているか？」「この段落はそのトークンコストに見合うか？」と問い直してください。

冗長な説明よりも、簡潔な例を優先してください。

### 適切な自由度の設定

タスクの脆弱性と変動性に合わせて、具体性のレベルを調整してください：

**高い自由度（テキストベースの指示）**：複数のアプローチが有効な場合、決定がコンテキストに依存する場合、またはヒューリスティックがアプローチを導く場合に使用します。

**中程度の自由度（パラメータ付きの擬似コードまたはスクリプト）**：推奨されるパターンが存在する場合、多少のバリエーションが許容される場合、または設定が動作に影響する場合に使用します。

**低い自由度（特定のスクリプト、少数のパラメータ）**：操作が脆弱でエラーが発生しやすい場合、一貫性が重要な場合、または特定の順序に従う必要がある場合に使用します。

Claudeが道を探索していると考えてください：崖のある狭い橋には具体的なガードレールが必要ですが（低い自由度）、開けた野原では多くのルートが可能です（高い自由度）。

### スキルの構造

すべてのスキルは、必須の `SKILL.md` ファイルと、オプションの整理されたリソースで構成されます：

```
skill-name/
├── SKILL.md (必須)
│   ├── YAMLフロントマターメタデータ (必須)
│   │   ├── name: (必須)
│   │   ├── description: (必須)
│   │   └── compatibility: (オプション、稀に必要)
│   └── Markdownによる指示 (必須)
└── 整理されたリソース (オプション)
    ├── scripts/          - 実行可能なコード (Python/Bashなど)
    ├── references/       - 必要に応じてコンテキストに読み込まれるドキュメント
    └── assets/           - 出力で使用されるファイル (テンプレート、アイコン、フォントなど)
```

#### SKILL.md (必須)

すべての `SKILL.md` は以下で構成されます：

- **フロントマター** (YAML): `name` と `description` フィールド（必須）、および `license`、`metadata`、`compatibility` などのオプションフィールドが含まれます。Claudeがスキルの起動を判断するために読み取るのは `name` と `description` のみであるため、スキルが何であるか、いつ使用すべきかを明確かつ包括的に記述してください。`compatibility` フィールドは環境要件（ターゲット製品、システムパッケージなど）を記載するためのものですが、ほとんどのスキルでは必要ありません。
- **本文** (Markdown): スキルを使用するための指示とガイダンス。スキルが起動した後にのみ読み込まれます。

#### 整理されたリソース (オプション)

##### スクリプト (`scripts/`)

決定論的な信頼性が必要なタスクや、繰り返し書き直されるタスクのための実行可能なコード（Python/Bashなど）。

- **含めるタイミング**: 同じコードが繰り返し書き直されている場合や、決定論的な信頼性が必要な場合
- **例**: PDF回転タスク用の `scripts/rotate_pdf.py`
- **利点**: トークン効率が良く、決定論的であり、コンテキストに読み込まずに実行できる場合がある
- **注意**: スクリプトも、パッチ適用や環境固有の調整のためにClaudeによって読み取られる必要がある場合があります

##### 参照資料 (`references/`)

Claudeのプロセスと言考を支援するために、必要に応じてコンテキストに読み込まれるドキュメントや参照資料。

- **含めるタイミング**: Claudeが作業中に参照すべきドキュメントがある場合
- **例**: 金融スキーマ用の `references/finance.md`、会社のNDAテンプレート用の `references/mnda.md`、会社ポリシー用の `references/policies.md`、API仕様用の `references/api_docs.md`
- **ユースケース**: データベーススキーマ、APIドキュメント、ドメイン知識、会社ポリシー、詳細なワークフローガイド
- **利点**: `SKILL.md` を軽量に保ち、Claudeが必要と判断したときにのみ読み込まれる
- **ベストプラクティス**: ファイルが大きい（10,000語以上）場合は、`SKILL.md` にgrep検索パターンを含めてください
- **重複を避ける**: 情報は `SKILL.md` または参照ファイルのいずれかに存在すべきであり、両方にあってはなりません。詳細な情報は、それがスキルの真の核心でない限り、参照ファイルに置くことを優先してください。これにより、コンテキストウィンドウを独占することなく、`SKILL.md` を軽量に保ちつつ情報を発見可能にできます。`SKILL.md` には主要な手続き的指示とワークフローガイダンスのみを保持し、詳細な参照資料、スキーマ、例などは参照ファイルに移動してください。

##### アセット (`assets/`)

コンテキストに読み込まれることは想定されておらず、むしろClaudeが生成する出力内で使用されるファイル。

- **含めるタイミング**: スキルが最終的な出力で使用されるファイルを必要とする場合
- **例**: ブランドアセット用の `assets/logo.png`、PowerPointテンプレート用の `assets/slides.pptx`、HTML/Reactボイラープレート用の `assets/frontend-template/`、タイポグラフィ用の `assets/font.ttf`
- **ユースケース**: テンプレート、画像、アイコン、ボイラープレートコード、フォント、コピーまたは変更されるサンプルドキュメント
- **利点**: 出力リソースをドキュメントから分離し、Claudeがファイルをコンテキストに読み込むことなく使用できるようにする

#### スキルに含めてはいけないもの

スキルには、その機能を直接サポートする必須のファイルのみを含めるべきです。以下のような余分なドキュメントや補助ファイルを作成しないでください：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- など

スキルには、AIエージェントが当面の仕事を遂行するために必要な情報のみを含めるべきです。作成に至ったプロセスの補助的なコンテキスト、セットアップやテストの手順、ユーザー向けドキュメントなどは含めないでください。追加のドキュメントファイルを作成することは、乱雑さと混乱を招くだけです。

### 段階的開示 (Progressive Disclosure) の設計原則

スキルは、コンテキストを効率的に管理するために3レベルの読み込みシステムを使用します：

1. **メタデータ (name + description)** - 常にコンテキストに存在（約100語）
2. **SKILL.md 本文** - スキルが起動したとき（5,000語未満）
3. **整理されたリソース** - Claudeが必要としたとき（スクリプトはコンテキストウィンドウに読み込まずに実行できるため、実質無制限）

#### 段階的開示のパターン

コンテキストの肥大化を最小限に抑えるため、`SKILL.md` 本文は要点に絞り、500行未満に抑えてください。この制限に近づいたら、コンテンツを別のファイルに分割してください。コンテンツを他のファイルに分割する場合は、スキルの読者がそれらの存在と使用タイミングを知ることができるように、`SKILL.md` からそれらを参照し、読み込むタイミングを明確に説明することが非常に重要です。

**核心となる原則：** スキルが複数のバリエーション、フレームワーク、またはオプションをサポートしている場合は、コアとなるワークフローと選択ガイダンスのみを `SKILL.md` に保持してください。バリアント固有の詳細（パターン、例、設定）は、別の参照ファイルに移動してください。

**パターン 1: 参照付きのハイレベルガイド**

```markdown
# PDF処理

## クイックスタート

pdfplumberでテキストを抽出する：
[コード例]

## 高度な機能

- **フォーム入力**: 完全なガイドについては [FORMS.md](FORMS.md) を参照
- **APIリファレンス**: すべてのメソッドについては [REFERENCE.md](REFERENCE.md) を参照
- **例**: 一般的なパターンについては [EXAMPLES.md](EXAMPLES.md) を参照
```

Claudeは、必要になったときにのみ `FORMS.md`、`REFERENCE.md`、または `EXAMPLES.md` を読み込みます。

**パターン 2: ドメイン固有の整理**

複数のドメインを持つスキルの場合、無関係なコンテキストの読み込みを避けるために、コンテンツをドメインごとに整理します：

```
bigquery-skill/
├── SKILL.md (概要とナビゲーション)
└── reference/
    ├── finance.md (収益、請求メトリクス)
    ├── sales.md (案件、パイプライン)
    ├── product.md (API使用状況、機能)
    └── marketing.md (キャンペーン、アトリビューション)
```

ユーザーがセールスメトリクスについて尋ねたとき、Claudeは `sales.md` のみを読み取ります。

同様に、複数のフレームワークやバリアントをサポートするスキルの場合、バリアントごとに整理します：

```
cloud-deploy/
├── SKILL.md (ワークフロー + プロバイダー選択)
└── references/
    ├── aws.md (AWSデプロイメントパターン)
    ├── gcp.md (GCPデプロイメントパターン)
    └── azure.md (Azureデプロイメントパターン)
```

ユーザーがAWSを選択した場合、Claudeは `aws.md` のみを読み取ります。

**パターン 3: 条件付きの詳細**

基本的なコンテンツを表示し、高度なコンテンツにリンクします：

```markdown
# DOCX処理

## ドキュメントの作成

新しいドキュメントには docx-js を使用してください。[DOCX-JS.md](DOCX-JS.md) を参照。

## ドキュメントの編集

単純な編集の場合は、XMLを直接変更してください。

**変更履歴について**: [REDLINING.md](REDLINING.md) を参照
**OOXMLの詳細について**: [OOXML.md](OOXML.md) を参照
```

Claudeは、ユーザーがそれらの機能を必要としたときにのみ `REDLINING.md` または `OOXML.md` を読み取ります。

**重要なガイドライン:**

- **深い階層の参照を避ける** - 参照ファイルは `SKILL.md` から1レベル深い場所に保持してください。すべての参照ファイルは `SKILL.md` から直接リンクされている必要があります。
- **長い参照ファイルの構造化** - 100行を超えるファイルの場合は、プレビュー時にClaudeが全容を把握できるように、上部に目次を含めてください。

## スキル作成プロセス

スキル作成には以下のステップが含まれます：

1. 具体的な例を用いてスキルを理解する
2. 再利用可能なスキルコンテンツ（スクリプト、参照資料、アセット）を計画する
3. スキルを初期化する (`init_skill.py` を実行)
4. スキルを編集する（リソースの実装と `SKILL.md` の執筆）
5. スキルをパッケージ化する (`package_skill.py` を実行)
6. 実際の使用に基づいて反復する

適用できない明確な理由がある場合を除き、これらのステップを順番に実行してください。

### ステップ 1: 具体的な例を用いたスキルの理解

スキルの使用パターンがすでに明確に理解されている場合にのみ、このステップをスキップしてください。既存のスキルを扱う場合でも、このステップは価値を持ち続けます。

効果的なスキルを作成するには、そのスキルがどのように使用されるかという具体的な例を明確に理解する必要があります。この理解は、ユーザーからの直接的な例、またはユーザーフィードバックによって検証された生成された例のいずれかから得られます。

例えば、`image-editor` スキルを構築する場合、関連する質問には以下が含まれます：

- 「image-editor スキルはどのような機能をサポートすべきですか？ 編集、回転、その他は？」
- 「このスキルがどのように使用されるかの例をいくつか挙げてもらえますか？」
- 「ユーザーが『この画像から赤目を削除して』や『この画像を回転させて』といったことを依頼するのが想像できます。他にどのような使用方法が考えられますか？」
- 「ユーザーがどのようなことを言ったときに、このスキルが起動されるべきですか？」

ユーザーを圧倒しないように、1つのメッセージで質問しすぎないようにしてください。最も重要な質問から始め、効果を高めるために必要に応じてフォローアップしてください。

スキルがサポートすべき機能が明確になった時点で、このステップを終了します。

### ステップ 2: 再利用可能なスキルコンテンツの計画

具体的な例を効果的なスキルに変えるために、各例を以下のように分析します：

1. 例の内容を一から実行する方法を検討する
2. これらのワークフローを繰り返し実行する際に、どのようなスクリプト、参照資料、アセットが役立つかを特定する

例： 「このPDFを回転させるのを手伝って」というリクエストを処理する `pdf-editor` スキルを構築する場合、分析の結果：

1. PDFの回転には、毎回同じコードを書き直す必要がある
2. `scripts/rotate_pdf.py` スリプトをスキルに保存しておくと役立つ

例： 「ToDoアプリを作って」や「歩数を追跡するダッシュボードを作って」というリクエストに対する `frontend-webapp-builder` スキルを設計する場合、分析の結果：

1. フロントエンドWebアプリの作成には、毎回同じボイラープレートのHTML/Reactが必要である
2. ボイラープレートのHTML/Reactプロジェクトファイルを含む `assets/hello-world/` テンプレートをスキルに保存しておくと役立つ

例： 「今日ログインしたユーザーは何人ですか？」というクエリを処理する `big-query` スキルを構築する場合、分析の結果：

1. BigQueryへのクエリ実行には、毎回テーブルスキーマとリレーションシップを再確認する必要がある
2. テーブルスキーマを文書化した `references/schema.md` ファイルをスキルに保存しておくと役立つ

スキルの内容を確立するために、各具体的な例を分析して、含めるべき再利用可能なリソース（スクリプト、参照資料、アセット）のリストを作成します。

### ステップ 3: スキルの初期化

この時点で、実際にスキルを作成します。

開発中のスキルがすでに存在し、反復またはパッケージ化が必要な場合にのみ、このステップをスキップしてください。その場合は、次のステップに進んでください。

新しいスキルを一から作成する場合は、常に `init_skill.py` スクリプトを実行してください。このスクリプトは、スキルが必要とするすべてを自動的に含む新しいテンプレートスキルディレクトリを便利に生成し、スキル作成プロセスをはるかに効率的で信頼性の高いものにします。

使用法：

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

スクリプトは：

- 指定されたパスにスキルディレクトリを作成します
- 適切なフロントマターとTODOプレースホルダーを含む `SKILL.md` テンプレートを生成します
- 例のリソースディレクトリ `scripts/`、`references/`、および `assets/` を作成します
- カスタマイズまたは削除可能な例のファイルを各ディレクトリに追加します

初期化後、生成された `SKILL.md` や例のファイルを必要に応じてカスタマイズまたは削除してください。

### ステップ 4: スキルの編集

（新しく生成された、または既存の）スキルを編集する際は、そのスキルがClaudeの別のインスタンスによって使用されるものであることを忘れないでください。Claudeにとって有益で非自明な情報を含めてください。どのような手続き的知識、ドメイン固有の詳細、または再利用可能なアセットが、別のClaudeインスタンスがこれらのタスクをより効果的に実行するのに役立つかを検討してください。

#### 実証済みの設計パターンの学習

スキルのニーズに基づいて、以下の役立つガイドを参照してください：

- **マルチステッププロセス**: シーケンシャルなワークフローと条件分岐ロジックについては `references/workflows.md` を参照
- **特定の出力形式や品質基準**: テンプレートや例のパターンについては `references/output-patterns.md` を参照

これらのファイルには、効果的なスキル設計のための確立されたベストプラクティスが含まれています。

#### 再利用可能なスキルコンテンツから始める

実装を開始するには、上記で特定した再利用可能なリソース（`scripts/`、`references/`、および `assets/` ファイル）から始めてください。このステップではユーザーの入力が必要になる場合があることに注意してください。例えば、`brand-guidelines` スキルを実装する場合、ユーザーは `assets/` に保存するブランドアセットやテンプレート、または `references/` に保存するドキュメントを提供する必要があるかもしれません。

追加されたスクリプトは、実際に実行してバグがないこと、および出力が期待通りであることをテストする必要があります。似たようなスクリプトがたくさんある場合は、完了までの時間のバランスを取りつつ、すべてが動作するという確信を得るために、代表的なサンプルのみをテストすれば十分です。

スキルに不要な例のファイルやディレクトリは削除してください。初期化スクリプトは、構造を示すために `scripts/`、`references/`、および `assets/` に例のファイルを作成しますが、ほとんどのスキルはそれらすべてを必要としません。

#### SKILL.md の更新

**執筆ガイドライン：** 常に命令形または不定詞の形式を使用してください。

##### フロントマター

`name` と `description` を使用してYAMLフロントマターを記述します：

- `name`: スキル名
- `description`: これはスキルの主要な起動メカニズムであり、Claudeがいつスキルを使用すべきかを理解するのに役立ちます。
  - スキルが何をするかと、それを使用する際の特定のトリガー/コンテキストの両方を含めてください。
  - すべての「いつ使用するか」の情報はここに含めてください。本文には含めないでください。本文は起動後にのみ読み込まれるため、本文内の「このスキルの使用タイミング」セクションはClaudeにとって役立ちません。
  - `docx`スキルの説明例： "変更履歴、コメント、書式の保持、およびテキスト抽出のサポートを備えた包括的なドキュメントの作成、編集、および分析。Claudeがプロフェッショナルなドキュメント（.docxファイル）を扱う必要がある場合に使用します： (1) 新しいドキュメントの作成、 (2) コンテンツの修正または編集、 (3) 変更履歴の処理、 (4) コメントの追加、またはその他のドキュメントタスク"

YAMLフロントマターには他のフィールドを含めないでください。

##### 本文

スキルとその整理されたリソースを使用するための指示を記述します。

### ステップ 5: スキルのパッケージ化

スキルの開発が完了したら、ユーザーと共有される配布可能な `.skill` ファイルにパッケージ化する必要があります。パッケージ化プロセスでは、スキルがすべての要件を満たしていることを確認するために、まず自動的に検証が行われます：

```bash
scripts/package_skill.py <path/to/skill-folder>
```

オプションの出力ディレクトリ指定：

```bash
scripts/package_skill.py <path/to/skill-folder> ./dist
```

パッケージ化スクリプトは以下を行います：

1. **検証**（自動）：以下の項目をチェックします：
   - YAMLフロントマターの形式と必須フィールド
   - スキルの命名規則とディレクトリ構造
   - 説明の完全性と品質
   - ファイル構成とリソース参照

2. **パッケージ化**：検証に合格した場合、スキルにちなんで名付けられた `.skill` ファイル（例： `my-skill.skill`）を作成します。これにはすべてのファイルが含まれ、配布に適したディレクトリ構造が維持されます。`.skill` ファイルは、拡張子が `.skill` のzipファイルです。

検証に失敗した場合、スクリプトはエラーを報告し、パッケージを作成せずに終了します。検証エラーを修正し、パッケージ化コマンドを再度実行してください。

### ステップ 6: 反復

スキルをテストした後、ユーザーから改善を求められることがあります。多くの場合、これはスキルを使用した直後、スキルがどのように機能したかという新鮮なコンテキストがあるずに行われます。

**反復ワークフロー：**

1. 実際のタスクでスキルを使用する
2. 苦労した点や非効率な点に気づく
3. `SKILL.md` または整理されたリソースをどのように更新すべきかを特定する
4. 変更を実装し、再度テストする
