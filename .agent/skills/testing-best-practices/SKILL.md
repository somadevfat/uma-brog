# テストベストプラクティス (Testing Best Practices)

このスキルは、プロジェクトにおけるテストの品質を最高水準に保つためのガイドラインを提供します。
「とりあえず通るテスト」ではなく、「仕様を網羅し、変更に対する自信を与えるテスト」を作成します。

## 1. コア原則

1.  **カバレッジ100%を基準とする**
    - 正常系だけでなく、異常系、境界値、条件分岐（if/else, switch, 三項演算子等）のすべてを網羅します。
    - カバレッジを達成すること自体が目的ではなく、すべてのコードパスが意図通りであることを証明するために100%を目指します。
2.  **意味のあるテスト (Meaningful Tests)**
    - 単なる関数の実行確認ではなく、ビジネスロジックやドメインの制約を忠実に検証します。
    - エッジケース（空配列、null/undefined、最大値、無効な入力など）を積極的に含めます。
3.  **AAA (Arrange, Act, Assert) パターンの遵守**
    - テストの構造を明確にするため、必ずセクション分けを行い、指定のコメントを記述します。

## 2. テストの構造 (AAAパターン)

すべてのテストケース (`it` または `test`) は、以下の構造に従ってください。

```typescript
test('テスト対象：期待される動作の日本語記述', async () => {
  // ## Arrange ##
  // テストデータの準備、モックの設定、依存オブジェクトの初期化など。

  // ## Act ##
  // テスト対象のメソッドや関数を実行。

  // ## Assert ##
  // 結果の検証。expect() を用いて期待値と比較。
});
```

## 3. 日本語記述ルールとドキュメント

プロジェクトの憲法に従い、すべて日本語で記述します。

-   **テスト名**: 実行結果や期待される振る舞いが一目でわかる日本語で記述します。
    -   良い例: `blogService.getPostById()：存在しないIDを指定した場合にNotFoundエラーをスローすること`
-   **JSDoc**: すべてのテストファイル、ユーティリティ、ヘルパーに関し、日本語で `/** ... */` 形式の記述を行います。
-   **セクションコメント**: 実装ひとまとまりを1行で説明します（AAAコメント以外の補助的な説明）。
-   **any禁止**: テストコードにおいても `any` 型の使用は禁止です。適切な型定義または `unknown` へのキャストを使用してください。

## 4. テスト設計のチェックリスト

テストを書く際、以下の項目を検討しているか確認してください。

- [ ] すべての条件分岐 (if, else, ternary) をカバーしているか？
- [ ] 境界値 (最小値、最大値、空文字、0など) をテストしているか？
- [ ] 外部APIやデータベースの失敗ケースをモックで再現しているか？
- [ ] 非同期処理において競合状態やタイムアウトを考慮しているか？
- [ ] テストデータが他のテストに影響を与えないよう、クリーンアップ (afterEach) を行っているか？
